# AI-SDD Manifesto

> **Version:** 1.0
>
> **Purpose:** Define the philosophy, values and engineering principles of the AI-SDD (Artificial Intelligence Spec-Driven Development) Framework.
>
> This manifesto is the foundation upon which every rule, template, prompt and workflow of the framework is built.
>
> It is intentionally stable. Technologies will evolve. These principles should endure.

---

# Introduction

Artificial Intelligence is transforming software engineering.

Large Language Models can generate code faster than any developer, but speed without discipline produces technical debt at an unprecedented scale.

AI-SDD was created from a simple observation:

> **The quality of software generated by AI is determined far more by the quality of its specifications than by the quality of its prompts.**

For decades, software engineering has developed mature practices for requirements, architecture, testing and documentation.

AI does not replace those practices.

It makes them more valuable.

AI-SDD is therefore not a prompt library.

It is not an IDE extension.

It is not tied to any specific language, framework or Large Language Model.

AI-SDD is an engineering methodology that places Specifications at the center of AI-assisted software development.

---

# Core Belief

Software should never begin with code.

Every software system begins with understanding.

Understanding produces Requirements.

Requirements produce Design.

Design produces Implementation.

Implementation produces Tests.

Tests produce Confidence.

Confidence produces Maintainable Software.

Changing this order inevitably increases uncertainty.

---

# Principle 1 — Solve Problems Before Writing Code

Software exists to solve real problems.

Technology is never the objective.

Every feature should begin by understanding:

* the problem,
* the users,
* the business value,
* and the expected outcome.

Code is only one possible solution.

---

# Principle 2 — Specifications Are the Source of Truth

Specifications define the expected behaviour of a system.

Code is an implementation of those specifications.

Whenever code and specifications disagree, the specification must be reviewed before changing either.

Specifications are first-class engineering artifacts.

---

# Principle 3 — Think Before Building

Good engineering is primarily a thinking activity.

Programming is only one stage of engineering.

The cost of architectural thinking is small.

The cost of architectural mistakes is enormous.

Time invested in analysis is rarely wasted.

---

# Principle 4 — One Artifact, One Responsibility

Every engineering artifact should have a single purpose.

Features describe opportunities.

Requirements describe behaviour.

Design describes structure.

Tasks describe implementation work.

Tests validate behaviour.

Reviews evaluate quality.

Releases communicate delivery.

Mixing responsibilities increases ambiguity for both humans and AI.

---

# Principle 5 — Simplicity Is a Feature

Complex systems fail in complex ways.

Simple systems are easier to understand, maintain and evolve.

AI-SDD values:

* simplicity over cleverness,
* clarity over abstraction,
* readability over optimization,
* maintainability over novelty.

Engineering excellence is often the removal of unnecessary complexity.

---

# Principle 6 — Traceability Is Mandatory

Every engineering decision should have a reason.

Every implementation should be traceable back to a Requirement.

Every Requirement should trace back to a Feature.

Every Test should validate one or more Requirements.

Every Review should explain its conclusions.

Traceability transforms software from isolated code into an understandable engineering system.

---

# Principle 7 — Architecture Exists to Reduce Complexity

Architecture is not documentation for its own sake.

Architecture exists to organize complexity.

Every architectural decision should make the system easier to understand, easier to change and easier to operate.

A successful architecture often contains fewer components than originally imagined.

---

# Principle 8 — Documentation Is Engineering

Documentation is not an afterthought.

Documentation captures decisions, assumptions and knowledge that code alone cannot express.

Well-written documentation reduces ambiguity.

Reduced ambiguity improves collaboration between humans and AI.

Documentation is therefore an engineering deliverable.

---

# Principle 9 — AI Is an Engineering Partner

Artificial Intelligence is an accelerator.

It is not an autonomous engineer.

AI generates possibilities.

Engineers make decisions.

Responsibility always remains with the engineering team.

Human judgement cannot be delegated.

---

# Principle 10 — Structure Improves AI

Language Models reason more effectively when information is structured.

Consistent documents.

Consistent terminology.

Consistent workflows.

Consistent identifiers.

These are not bureaucratic constraints.

They are mechanisms that improve reasoning quality.

AI performs best inside well-designed systems.

---

# Principle 11 — Reusable Frameworks Outlive Prompts

Prompts are tactical.

Frameworks are strategic.

A good prompt solves today's problem.

A good engineering framework improves every future project.

AI-SDD prioritizes reusable engineering practices over isolated prompt engineering.

---

# Principle 12 — Engineering Decisions Should Be Explicit

Hidden assumptions create future defects.

Important decisions should always be documented.

Rejected alternatives should be remembered.

Trade-offs should be acknowledged.

Future engineers should understand not only what was decided, but why.

---

# Principle 13 — Quality Is Designed, Not Inspected

Quality cannot be added at the end of a project.

Requirements influence quality.

Architecture influences quality.

Implementation influences quality.

Testing verifies quality.

Reviews improve quality.

Every engineering phase contributes to the final result.

---

# Principle 14 — Testing Validates Behaviour

Tests exist to verify that software satisfies its Requirements.

Testing should validate observable behaviour.

Passing tests do not guarantee good software.

Failing tests always indicate uncertainty.

Confidence grows through verification.

---

# Principle 15 — Consistency Enables Automation

Predictable engineering artifacts allow predictable AI behaviour.

Consistent templates.

Consistent workflows.

Consistent terminology.

Consistent document structure.

Automation depends upon consistency.

---

# Principle 16 — Tools Are Temporary

Programming languages change.

Frameworks change.

Editors change.

Models change.

Hardware changes.

Engineering principles evolve far more slowly.

AI-SDD intentionally separates methodology from tooling.

The framework should remain useful regardless of whether development is performed using Continue, VS Code, Ollama, cloud models or future AI systems.

---

# Principle 17 — Continuous Improvement Without Constant Reinvention

Engineering frameworks should evolve deliberately.

Small improvements accumulate.

Constant redesign destroys stability.

AI-SDD values evolutionary improvement over perpetual reinvention.

Stability is a feature.

---

# Principle 18 — Engineering Is Communication

Software engineering is fundamentally a communication discipline.

Requirements communicate intent.

Design communicates structure.

Code communicates implementation.

Tests communicate expected behaviour.

Documentation communicates knowledge.

AI-SDD extends this principle by recognising that engineering artifacts must communicate effectively with both humans and Artificial Intelligence.

---

# Final Principle

The objective of AI-SDD is not to generate more code.

The objective is to generate **better software**.

Better software emerges from better understanding.

Better understanding emerges from better specifications.

Better specifications emerge from disciplined engineering.

Artificial Intelligence accelerates the journey.

Engineering determines the destination.
